> **版本**: v1.0

> **目的**: 定义事件分级、响应流程、升级路径和复盘机制

---

## 一、事件分级 (Severity Levels)

| 级别 | 名称 | 定义 | 响应时间 | 示例 |
|------|------|------|---------|------|
| **S0** | 严重 | 产品完全不可用，核心功能失效，影响所有用户 | 立即 (15min) | App无法启动、数据丢失、严重安全漏洞 |
| **S1** | 高 | 核心功能受限，影响主要用户场景 | 30min | 登录失败、支付中断、CI系统宕机 |
| **S2** | 中 | 非核心功能异常，部分用户受影响 | 2h | 统计报表延迟、通知失败、非关键页面报错 |
| **S3** | 低 | 轻微问题或cosmetic问题 | 24h | 文字错误、UI对齐问题、非关键路径提示信息 |

### 严重性判断树

```
用户报告的问题
    │
    ├─ 是否影响核心业务流程？
    │   ├─ YES → S1 或 S0
    │   └─ NO → 继续判断
    │
    ├─ 影响用户比例？
    │   ├─ >50% → S1
    │   ├─ 10-50% → S2
    │   └─ <10% → S3
    │
    └─ 是否有workaround？
        ├─ YES → 可降级
        └─ NO → 需升级
```

## 二、响应流程

### 2.1 S0 响应流程 (战争状态)

```
发现/报告 S0
    │
    ├─ 立即通知 → 所有人 (飞书群/电话)
    │
    ├─ 组建 incident channel → #incident-YYYY-MM-DD
    │
    ├─ Incident Commander (IC) 上任 → 负责协调
    │   └─ IC = 发现者或值班 on-call
    │
    ├─ 所有人立即停止新功能开发 → 聚焦恢复
    │
    ├─ 每 15min 更新状态
    │
    └─ 恢复后 → 48h 内完成复盘
```

### 2.2 S1 响应流程

```
发现/报告 S1
    │
    ├─ 通知 → 值班 on-call + CEO
    │
    ├─ 创建 #incident-YYYY-MM-DD channel
    │
    ├─ 30min 内给出初步评估
    │
    ├─ 每 30min 更新状态
    │
    └─ 恢复后 → 72h 内完成复盘
```

### 2.3 S2-S3 响应流程

```
发现/报告 S2/S3
    │
    ├─ 创建 JIRA/Issue 记录
    │
    ├─ 下一个工作日处理
    │
    └─ 每周汇总review
```

## 三、角色定义

### 3.1 Incident Commander (IC)
- **职责**: 协调、沟通、决策
- **权限**: 召集任何人、暂停任何非紧急任务
- **当前**: CEO (周小墨) 担任

### 3.2 Technical Lead
- **职责**: 技术调查、根因分析、修复执行
- **来源**: Developer Agent / 人工开发者

### 3.3 Communications Lead
- **职责**: 状态更新、用户沟通、复盘文档
- **来源**: CEO 或指定人员

## 四、升级路径

```
Level 1: 发现者
  └─ 自己能解决？ → YES: 修复
                   NO: 升级

Level 2: 值班 on-call
  └─ 30min 内能解决？ → YES: 修复
                        NO: 升级

Level 3: IC + Technical Lead
  └─ 2h 内能解决？ → YES: 修复
                    NO: 升级 (外部资源/更大团队)

Level 4: 外部资源
  - Flutter 社区
  - Firebase 支持
  - App Store Review Board
```

### 升级触发条件

| 条件 | 触发升级 |
|------|---------|
| 30min 无进展 | → 升级到 Level 3 |
| 2h 未恢复 | → 升级到 Level 4 |
| 数据泄露怀疑 | → 立即升级到 Level 4 |
| 涉及第三方服务 | → 通知第三方，同时内部继续 |

## 五、常见事故类型

### 5.1 App 崩溃 (S0-S1)

```
迹象:
- Crashlytics 报警
- App Store 投诉增加
- 用户反馈 "打不开"

响应:
1. 检查 Crashlytics 仪表板
2. 确定 crash 率 > 1% → S1
3. 确定 crash 率 > 5% 或 top crash → S0
4. 确定受影响版本范围
5. 回滚 (如果刚发布) 或热修复
```

### 5.2 CI/CD 故障 (S1-S2)

```
迹象:
- GitHub Actions 失败
- 自动化测试失败
- 部署管道中断

响应:
1. 检查 GitHub Actions 日志
2. 确定是 flaky test 还是 real bug
3. Flaky: 重试 + 标记
4. Real: 修复后重新运行
```

### 5.3 数据问题 (S0-S1)

```
迹象:
- 数据不一致
- 数据丢失
- 数据泄露

响应:
1. 立即停止数据写入 → 防止扩大
2. 评估影响范围
3. 恢复备份 (如果有)
4. 通知受影响的用户
5. 上报监管 (如果需要)
```

### 5.4 安全漏洞 (S0)

```
迹象:
- 渗透测试发现
- 异常访问模式
- 用户报告

响应:
1. 立即隔离受影响系统
2. 评估暴露面
3. 修复漏洞
4. 检查日志确认是否被利用
5. 如果已利用 → 法律/合规团队介入
```

## 六、状态更新模板

### 6.1 初始状态 (5min 内)

```
🚨 [S0/S1] 事件标题
影响: 描述影响范围
状态:调查中
预计恢复: [TBD]
负责人: @IC
```

### 6.2 进度更新 (每 15/30min)

```
📊 [S0/S1] 事件标题
状态: [调查中/已定位/修复中/恢复中]
进展: 发现了什么
下一步: 正在做什么
预计恢复: [时间]
```

### 6.3 恢复通知

```
✅ [已恢复] 事件标题
恢复时间: YYYY-MM-DD HH:MM
持续时间: Xh Xm
影响: 描述
原因: 简要说明
后续: 需要做什么防止复发
```

## 八、预防措施

### 8.1 发布检查清单

```
发布前:
☐ 功能测试通过
☐ 自动化测试覆盖率 > 80%
☐ Code Review 完成
☐ 安全扫描通过
☐ 性能测试通过
☐ 回滚方案准备

发布中:
☐ 灰度发布 (10% → 50% → 100%)
☐ 监控仪表板实时观察
☐ Crashlytics 实时监控
☐ 准备随时回滚

发布后:
☐ 30min 内检查核心指标
☐ 1h 后确认无异常
☐ 24h 后确认稳定性
```

### 8.2 监控指标

| 指标 | 阈值 | 告警 |
|------|------|------|
| Crash Rate | > 1% | S1 |
| Crash Rate | > 5% | S0 |
| API Error Rate | > 1% | S2 |
| API Latency P99 | > 2s | S2 |
| CI Pipeline Success | < 95% | S2 |
