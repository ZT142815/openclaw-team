## 三、工作流程

### 3.1 任务开发流程

```
接收任务（从CEO/产品）
    ↓
理解需求（有问题立即沟通）
    ↓
技术方案设计（复杂功能）
├── 架构设计
├── 数据库设计
└── 接口设计
    ↓
代码开发
├── 按规范开发
├── 编写注释
└── 单元测试
    ↓
自测验证
├── 功能测试
├── 边界测试
└── 异常测试
    ↓
提交Code Review（CEO）
    ↓
CEO Review
├── 检查代码质量
├── 检查安全性
└── 检查性能
    ↓
修复问题（如有）
    ↓
合并到主分支
```

### 3.2 技术方案设计流程

```
复杂功能开发前
    ↓
技术方案评审
├── 方案设计
├── 备选方案对比
├── 风险评估
└── 资源评估
    ↓
评审会议（产品+研发+CEO）
    ↓
确定最终方案
    ↓
按方案开发
```

### 3.3 Bug修复流程

```
收到Bug报告
    ↓
初步定位
├── 复现Bug
├── 定位问题代码
└── 分析根因
    ↓
修复Bug
├── 编写修复代码
└── 添加回归测试
    ↓
自测验证
    ↓
提交Review
    ↓
CEO Review
    ↓
合并发布
```

## 五、代码规范

### 5.1 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 变量 | 小驼峰 | userName |
| 常量 | 全大写 | MAX_COUNT |
| 函数 | 小驼峰 | getUser() |
| 类名 | 大驼峰 | UserService |
| 文件 | 下划线 | user_service.dart |

### 5.2 Git提交规范

```
格式: type: description

type:
- feat: 新功能
- fix: 修复bug
- docs: 文档
- style: 格式
- refactor: 重构
- test: 测试
- chore: 构建/工具
```

### 5.3 Code Review检查清单

| 检查项 | 说明 |
|--------|------|
| 功能正确性 | 符合需求 |
| 代码规范 | 命名/格式 |
| 安全性 | 无注入/泄露 |
| 性能 | 无性能问题 |
| 测试覆盖 | 核心有测试 |
