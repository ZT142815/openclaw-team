## 一、为什么需要Code Review

```
没有Code Review的风险：
- Bug隐藏在代码中
- 代码风格不一致
- 安全隐患
- 技术债务累积
```

---

## 二、Code Review检查清单

### 2.1 功能正确性

| 检查项 | 说明 |
|--------|------|
| 代码逻辑正确 | 实现符合需求 |
| 边界条件处理 | 空值、异常情况 |
| 业务规则 | 符合业务逻辑 |

### 2.2 代码质量

| 检查项 | 说明 |
|--------|------|
| 命名规范 | 变量/函数/类名清晰 |
| 代码简洁 | 不重复、无冗余 |
| 函数长度 | 单个函数不超过50行 |
| 注释 | 复杂逻辑有解释 |

### 2.3 安全性

| 检查项 | 说明 |
|--------|------|
| SQL注入 | 使用参数化查询 |
| XSS | 输出转义 |
| 敏感信息 | 不硬编码密码/密钥 |
| 权限校验 | 接口有权限验证 |

### 2.4 性能

| 检查项 | 说明 |
|--------|------|
| 数据库查询 | 避免N+1查询 |
| 循环优化 | 避免嵌套循环 |
| 缓存 | 合理使用缓存 |
| 懒加载 | 按需加载 |

### 2.5 测试覆盖

| 检查项 | 说明 |
|--------|------|
| 单元测试 | 核心函数有测试 |
| 测试质量 | 测试真实场景 |

## 三、Review流程

### 3.1 CEO Review流程

```
开发Agent完成代码
    ↓
提交到待Review队列
    ↓
CEO收到Review请求
    ↓
CEO审查代码
    ↓
反馈意见（通过/需要修改）
    ↓
开发Agent修复问题
    ↓
CEO再次Review
    ↓
通过后合并
```

### 3.2 CEO Review职责

| 职责 | 说明 |
|------|------|
| 功能正确性 | 代码逻辑是否符合需求 |
| 代码质量 | 命名、注释、结构 |
| 架构合理 | 是否符合技术方案 |
| 安全检查 | 是否有安全隐患 |
| 性能意识 | 是否有性能问题 |

### 3.3 Review标准

| 评级 | 说明 |
|------|------|
| ⭐⭐⭐⭐⭐ | 完美，可以直接合并 |
| ⭐⭐⭐⭐ | 良好，有小问题需要修复 |
| ⭐⭐⭐ | 一般，有较大问题必须修复 |
| ⭐⭐ | 差，需要重写 |
| ⭐ | 不可接受 |

## 五、分支管理策略

### 5.1 分支命名规范

```
master            # 正式环境代码
develop           # 开发环境代码
feature/{功能名}  # 功能分支
bugfix/{Bug编号}  # Bug修复分支
sprint/{Sprint号} # Sprint分支
release/{版本号}  # 发布分支
hotfix/{问题}     # 热修复分支
```

### 5.2 分支流程

```
feature/登录功能
    ↓ 完成
    ↓ PR
develop
    ↓ Sprint结束
    ↓ PR
release/v1.0.0
    ↓ 测试通过
    ↓ PR
master (打tag)
```

## 六、我们的Code Review机制

### 6.1 职责分工

| 角色 | 职责 |
|------|------|
| **CEO** | 最终Review决策、架构把控、安全审查 |
| **开发Agent** | 自检、修复、提交 |
| **自动化工具** | 辅助检查（flutter analyze等）|

### 6.2 Review请求格式

```markdown
## Code Review请求

### 项目
{项目名称}

### 分支
{分支名称}

### 功能描述
{简要描述这次代码实现了什么}

### 自检结果
- [x] flutter analyze 通过
- [x] 功能测试通过
- [ ] 有已知问题需要CEO判断

### 等待CEO确认
```

### 6.3 CEO Review输出格式

```markdown
## CEO Code Review

### 项目: {项目名称}
### 日期: {YYYY-MM-DD HH:MM}
### 评分: ⭐⭐⭐⭐⭐

#### 总体评价
{总体评价}

#### 发现的问题
1. {问题1}
2. {问题2}

#### 结论
- [ ] 可以合并
- [ ] 需要修复后再次Review

#### 意见
{具体改进建议}
```
